014/**
015 * Provides static methods to encode Coords as single primitive ints in various ways, hence the namesake, but also
016 * provides advanced methods to encode 2D arrays of various sorts produced by SquidLib in extremely memory-efficient
017 * representations, and decode those representations to various types of 2D array on-demand. There's a detailed
018 * introduction on the SquidLib wiki, https://github.com/SquidPony/SquidLib/wiki/Handling-Map-Regions-with-CoordPacker ,
019 * which is probably the best way to learn the techniques possible with this class. Most methods in this aren't useful
020 * on their own, but can be mixed and matched to get specific regions from a map, such as all floors not adjacent to a
021 * wall, or all grass within 3 squares of deep or shallow water, with walls blocking the distance measurement. You can
022 * also use packed data that this class produces as keys for {@link RegionMap} to associate values with regions.
023 * <br>
024 * NOTE: Internally, this class is atypically complex and low-level for SquidLib because it is attempting to attain some
025 * very challenging performance gains. You should not consider it idiomatic SquidLib code or start modifying it unless
026 * you have a good grasp of bitwise operations and the performance implications, particularly in regard to memory
027 * consumption, that higher-level and more convenient Java programming techniques have.
028 * <br>
029 * NOTE 2: This class fills a role that is very similar to {@link GreasedRegion}, and there are times when code will do
030 * better using CoordPacker than GreasedRegion and vice versa. GreasedRegion uses objects and tries to change them
031 * in-place whenever possible instead of allocating new data; CoordPacker uses short[] values that it never changes
032 * in-place and frequently allocates new short[] values to represent data. GreasedRegion stores the full map without any
033 * compression beyond "one bit per cell" and always needs to store all cells on the map (sometimes slightly more);
034 * CoordPacker can compress some data to significantly less than one bit per cell, and doesn't care about the maximum
035 * bounds of the map as long as it fits inside a 256x256 square. GreasedRegion is significantly faster than CoordPacker
036 * when performing any operations that move, shrink, or expand an area, such as {@link #expand(short[], int, int, int)},
037 * {@link #retract(short[], int, int, int)}, and {@link #translate(short[], int, int, int, int)}. CoordPacker usually
038 * (not always) uses somewhat less memory than GreasedRegion, though both use very little unless the maps are very
039 * large or very numerous. Both can do essentially the same things, except that GreasedRegion has no equivalent to the
040 * {@link #packMulti(byte[][], int)} and other Multi kinds of method, no {@link #radiate(short[], short[], int)}, and no
041 * {@link #reachable(short[], short[], Reach)}, while CoordPacker has no equivalent to
042 * {@link GreasedRegion#expandSeriesToLimit()} and other ToLimit kinds of method, no
043 * {@link GreasedRegion#flip(int, int)}, no {@link GreasedRegion#deteriorate(RNG, int)}, no
044 * {@link GreasedRegion#insert(int, int, GreasedRegion)} (though CoordPacker can generally use
045 * {@link #unionPacked(short[], short[])} for that), and several other methods don't have equivalents. In general, you
046 * will probably find GreasedRegion more intuitive because it involves working with objects instead of a short[] that is
047 * treated like a particular kind of data, and the methods are also somewhat more clearly-named. GreasedRegion also
048 * implements Collection of Coord, while the short[] data can't implement anything. A way around this for CoordPacker is
049 * the {@link squidpony.squidgrid.zone.CoordPackerZone} class; both that and GreasedRegion implement the
050 * {@link squidpony.squidgrid.zone.Zone} interface, which ensures a way to get a List of Coord from the Zone.
051 * <br>
052 * The pack() methods in this class take a 2D array with a clear division between cells in an "on" state and cells in an
053 * "off" state, and they produce a very tightly compressed short array that can be losslessly decompressed with the
054 * unpack() methods to a boolean 2D array that stores equivalent on/off data to the input. The packMulti() method in
055 * this class takes a double 2D array that has more than two states that may need to be encoded, such as an FOV map that
056 * stores light level as a value between 0.0 and 1.0 instead of just on or off, and an additional double array that
057 * defines what states should be distinguished in the result (for example, if the FOV can store values that differ by
058 * 0.1 for a FOV radius of 10, you could pass the array of 10 levels: 0.1, 0.2, 0.3, ... 0.9, 1.0). The value returned
059 * by packMulti() is a short[][], but with different array lengths for each sub-array (a jagged array); the length of
060 * the short[][] is the same as the length of the levels array, and each sub-array corresponds to a different level of
061 * FOV lighting or other gradation as defined in levels. This short[][] can be passed to the unpackMultiByte() method in
062 * this class to produce a byte 2D array where the original levels correspond to progressively greater bytes, with 0
063 * used for cells that were less than the smallest value in levels, 1 for values that were only greater than the
064 * smallest value, and no others, in levels, then 2 for larger values, etc. until it places a byte with a value equal to
065 * the length of levels in the cells that are the highest. There is also the unpackMultiDouble() method in this class
066 * that takes the same short[][] unpackMultiByte() can take, but also takes a levels double array that should be the
067 * same as the one used to compress short[][]. It will return a double 2D array with any cells that were smaller than
068 * the smallest value in levels assigned 0.0, and any other cells will be assigned a double that corresponds to the
069 * highest value in levels that does not exceed the original double at that location in the unpacked data. To make this
070 * more clear, if you have 4 levels: [0.25, 0.5, 0.75, 1.0] and you packMulti() on an FOV with a large radius and
071 * sample values 0.1, 0.45, 0.8, 1.0, you will get a packed short[][] with 4 sub-arrays to match the 4 levels. If you
072 * then pass the short[][] and levels to unpackMultiDouble later, much of the same radius will be filled, but because
073 * the sample value 0.1 was less than the smallest value in levels, its cell will be given 0.0. What was originally 0.45
074 * will be given the next-lower levels value, 0.25; 0.8 will be given 0.75, and 1.0 will remain 1.0.
075 * <br>
076 * This compression is meant to produce a short[] or short[][] that uses as little memory as possible for the specific
077 * case of compressing maps with these qualities:
078 * <ul>
079 *     <li>Maps are not especially large for a grid-based game; the maximum size is 256x256 cells.</li>
080 *     <li>The vast majority of that 256x256 space is either unused or filled with cells no greater than 0.</li>
081 *     <li>The cells that are greater than 0 are mostly near each other, though separate areas are possible.</li>
082 * </ul>
083 * These properties are all shared by typical roguelike FOV maps, and the specificity of these circumstances mean
084 * extraordinarily dense compression can be achieved using the right combination of algorithms. In early testing,
085 * using dungeon maps generated by {@link squidpony.squidgrid.mapping.DungeonGenerator} that should be typical of
086 * roguelike maps and a diamond-shaped FOV with radius 8, compression of the short[] returned by pack() vs.
087 * the original double[][] (which wastefully represents 2 states with 8 bytes) yields average memory usage ratios
088 * between (with relatively optimal parameters) 0.0001237905030818498 in one of the best cases, and (with some very
089 * poor parameters for the dungeon, but still using a realistic FOV map) 0.003135985198889917 in one of the worst.

121 * In the best-case scenario of packing a 240x240 double array to a short array encoding two states, the result
122 * uses less than 1/8000 the memory that the input uses. Writing to disk can store both input and output more
123 * efficiently, but the method used here should ensure that even encoding the input FOV map as a flat sequence of
124 * single bits and compressing the file should still be on par with the output of pack() due to optimization to
125 * ensure nearby cells on a map are compressed together.
126 * <br>
127 * The technique used by this class is to walk along a Hilbert Curve, storing whether the walk is traveling through
128 * "on" or "off" cells, which can be determined by a comparison to a number or a boolean, then encoding alternate shorts
129 * into the short[] to be returned, with even-number indices (starting at 0) in the array corresponding to the number of
130 * contiguous cells walked through in the "off" state, and odd-number indices corresponding to the number of
131 * contiguous cells walked through in the "on" state. A user of this library does not need to understand the details
132 * and properties of this algorithm unless they want to generate maps that will compress more optimally. In short:
133 * <ul>
134 * <li>Smaller maps tend to be processed faster by pack(), since the nature of a Hilbert Curve means a map that
135 * fits in one half the width and one half the height of the curve only needs to walk one quarter of the Curve to
136 * get all the needed information.</li>
137 * <li>Smaller maps also compress less optimally ratio-wise than larger maps with the same area of "on" cells. The
138 * compression ratio approaches its best when using very large maps, such as 240x240, and encoding just a few
139 * cells on that map (such as for a small FOV radius or a cramped room). A map that is entirely "off" uses only 16
140 * bytes of RAM (the minimum for any array on the JVM).</li>
141 * <li>Unusually shaped maps can cause compression problems by forcing adjacent cells to sometimes require walking
142 * more cells than needed to get to an adjacent cell. For example, a map greater than 64 cells tall, but less than
143 * 33 cells wide, has properties that require walking through a large empty area to get to sometimes only a few
144 * cells that are "on" before it walks back through empty space. Similarly, a map that is greater than 128 cells
145 * tall but is otherwise narrow has the same property of requiring walking through empty space, but also requires
146 * the entire Curve to be walked even if the map's width is only a tiny fraction of the Curve's 256 cells.</li>
147 * </ul>
148 * <b>In shorter-than-short</b>, you'll get particularly good results for compression speed and compressed size with
149 * maps approximately these sizes: 240x240, 240x120, 120x120, 60x120, 60x60, 60x30, 30x30. The biggest maps have the
150 * best relative gain on compressed memory usage, and the smallest maps have the best compression speed.
151 *<br>
152 * The details of the algorithm are not terribly complex once you understand the Hilbert Curve. The simplified
153 * version of the Hilbert Curve that SquidLib employs is essentially a path through a square grid (it must have side
154 * lengths that are powers of 2, and SquidLib always uses 256), starting in the corner cell (x=0,y=0), ending in the
155 * corner cell (x=0,y=255), and traversing every other cell on the grid along its path without ever traveling in a
156 * loop, crossing the path it walked, or moving in any direction but one cell up, down, left, or right. The shape
157 * of the path this takes has the useful property of keeping most groups of cells walked through with similar x and
158 * y at similar distances traveled from the start of the curve, and most groups of cells with very dissimilar x and
159 * y at very different distances traveled. Since FOV and several other things you might want to encode with CoordPacker
160 * tends to be clustered in small areas and occupy more complicated shapes than straight lines due to dungeon layout
161 * blocking sections of FOV, the simplest paths of a wide zigzag from side-to-side, or an outward-going-in spiral, have
162 * rather poor behavior when determining how much of an area they pass through contiguously. The contiguous area trait
163 * is important because of the next step: Run-Length Encoding.
164 *<br>
165 * Run-Length Encoding is much simpler to explain than the Hilbert Curve, especially without visual aids. In the version
166 * SquidLib uses, only on or off states need to be recorded, so the method used here is smaller and more efficient than
167 * most methods that need to store repeated characters in strings (and letters, numbers, and punctuation clearly have
168 * more than 2 states). The technique works like this:
169 *<br>
170 * Start in the "off" state, walk down the Hilbert Curve counting how many cells you walk through that are still "off,"
171 * and when you encounter a cell that is "on," you write down how many cells were off, transition to the "on" state. Now
172 * keep walking the Hilbert Curve, but counting how many cells you walk through that are still "on." When you reach
173 * an "off" cell, write down how many were "on," then start walking and counting again, with your count starting at 0.
174 * Repeat until you reach the end of the Hilbert Curve, but if you reach the end while counting "off" cells, you don't
175 * need to write down that number (a shortcut allows many maps to stop sooner than the 65,536th element of the Curve).
176 *<br>
177 * There are some additional traits that relate to the edge of the map being treated as "off" even though no
178 * calculations are done for cells out of map bounds, and some optimizations that ensure that maps that are smaller than
179 * a half, a quarter, or an eighth of the 256x256 curve in both dimensions (and sometimes just one) only need to walk a
180 * portion of the Hilbert Curve and simply skip the rest without walking it.
181 *<br>
182 * The Hilbert Curve has not been definitively proven to be the best possible path to ensure 1D distance and 2D location
183 * are similar, but it has been extensively used for tasks that require similar locations for similar distances (in
184 * particular, it has become useful in supercomputing clusters for allocating related work to physically nearby
185 * machines), and since there hasn't been anything with better spatial properties discovered yet, this technique should
186 * remain useful for some time.

239    /**
240     * Compresses a double[][] (typically one generated by {@link squidpony.squidgrid.FOV}) that only stores two
241     * relevant states (one of which should be 0 or less, the other greater than 0), returning a short[] as described in
242     * the {@link CoordPacker} class documentation. This short[] can be passed to CoordPacker.unpack() to restore the
243     * relevant states and their positions as a boolean[][] (with false meaning 0 or less and true being any double
244     * greater than 0). As stated in the class documentation, the compressed result is intended to use as little memory
245     * as possible for most roguelike FOV maps. To avoid floating-point number comparison issues, this actually needs
246     * doubles to be greater than 0.0001, which should never cause incorrect behavior with FOV's double[][] maps.
247     * <br>
248     * <b>To store more than two states</b>, you should use packMulti().
249     *
250     * @param map a double[][] that probably was returned by FOV. If you obtained a double[][] from DijkstraMap, it
251     *            will not meaningfully compress with this method.
252     * @return a packed short[] that should, in most circumstances, be passed to unpack() when it needs to be used.
253     */
254    public static short[] pack(double[][] map)
255    {
256        if(map == null || map.length == 0)
257            throw new ArrayIndexOutOfBoundsException("CoordPacker.pack() must be given a non-empty array");
258        int xSize = map.length, ySize = map[0].length;
259        if(xSize > 256 || ySize > 256)
260            throw new UnsupportedOperationException("Map size is too large to efficiently pack, aborting");
261        ShortVLA packing = new ShortVLA(64);
262        boolean on = false, anyAdded = false, current;
263        int skip = 0, limit = 0x10000, mapLimit = xSize * ySize;
264        if(ySize <= 128) {
265            limit >>= 1;
266            if (xSize <= 128) {
267                limit >>= 1;
268                if (xSize <= 64) {
269                    limit >>= 1;
270                    if (ySize <= 64) {
271                        limit >>= 1;
272                        if (ySize <= 32) {
273                            limit >>= 1;
274                            if (xSize <= 32) {
275                                limit >>= 1;
276                            }
277                        }
278                    }
279                }
280            }
281        }
282
283        for(int i = 0, ml = 0; i < limit && ml < mapLimit; i++, skip++)
284        {
285            if(hilbertX[i] >= xSize || hilbertY[i] >= ySize) {
286                if(on) {
287                    on = false;
288                    packing.add((short) skip);
289                    skip = 0;
290                    anyAdded = true;
291                }
292                continue;
293            }
294            ml++;
295            current = map[hilbertX[i]][hilbertY[i]] > 0.0001;
296            if(current != on)
297            {
298                packing.add((short) skip);
299                skip = 0;
300                on = current;
301                anyAdded = true;
302            }
303        }
304        if(on)
305            packing.add((short)skip);
306        else if(!anyAdded)
307            return ALL_WALL;
308        return packing.toArray();
309    }

4669     * aforementioned data assuming a bare dungeon called map with WIDTH and HEIGHT constants using:
4670     * <br>

4672     * rooms = flood(floors, retract(floors, 1, WIDTH, HEIGHT, true), 2, false),
4673     * corridors = differencePacked(floors, rooms),
4674     * doors = intersectPacked(rooms, fringe(corridors, 1, WIDTH, HEIGHT, false));}
4675     * <br>
4676     * You can then get all rooms as separate regions with {@code List<short[]> apart = split(rooms);}, or substitute
4677     * {@code split(corridors)} to get the corridors. The room-finding technique works by shrinking floors by a radius
4678     * of 1 (8-way), which causes thin areas like corridors of 2 or less width to be removed, then flood-filling the
4679     * floors out from the area that produces by 2 cells (4-way this time) to restore the original size of non-corridor
4680     * areas (plus some extra to ensure odd shapes are kept). Corridors are obtained by removing the rooms from floors.
4681     * The example code also gets the doors (which overlap with rooms, not corridors) by finding where the a room and a
4682     * corridor are adjacent. This technique is used with some enhancements in the RoomFinder class.

4686     */

4689        ArrayList<short[]> arrays = new ArrayList<>(32);
4690        short[] remaining = new short[packed.length];

4693            boolean on = false;
4694            int idx = 0;
4695            for (int p = 0; p < remaining.length; p++, on = !on) {
4696                if (on) {
4697                    short[] area = flood(packed, packOne((short) idx), 512, false);
4698                    arrays.add(area);
4699                    remaining = differencePacked(remaining, area);
4700                    break;
4701                }
4702                idx += remaining[p] & 0xffff;
4703            }
4704        }
