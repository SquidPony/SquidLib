010/**
011 * Simple hashing functions that we can rely on staying the same cross-platform.
012 * The static methods of this class (not its inner classes) use the Fowler/Noll/Vo
013 * Hash (FNV-1a) algorithm, which is public domain. The hashes this returns are always
014 * 0 when given null to hash. Arrays with identical elements of identical types will
015 * hash identically. Arrays with identical numerical values but different types will
016 * hash differently (only for FNV-1a, not the inner classes). FNV-1a may be somewhat
017 * slow if you are running many hashes or hashing very large data; in that case you
018 * should consider one of the inner classes, Lightning or Storm.
019 * <br>
020 * There are several static inner classes in CrossHash, Lightning, Storm, Falcon, Chariot, Wisp,
021 * and Mist, that provide different hashing properties, as well as the inner IHasher interface.
022 * Of these, Wisp should be the default choice in most cases. If you need a salt to alter the
023 * hash function, using one of a large family of such functions instead of a single function like
024 * Wisp, then Storm and Mist are good choices (Mist is faster, Storm is more stable API-wise).
025 * Falcon and Lightning are mostly superseded by Wisp. Chariot has some limitations, and uses
026 * potentially much more memory than the others (which use almost none).
027 * <br>
028 * IHasher values are provided as static fields, and use Wisp to hash a specific type or fall
029 * back to Object.hashCode if given an object with the wrong type. IHasher values are optional
030 * parts of OrderedMap and OrderedSet, and allow arrays to be used as keys in those collections
031 * while keeping hashing by value instead of the normal hashing by reference for arrays. You
032 * probably won't ever need to make a class that implements IHasher yourself; for some cases you
033 * may want to look at the {@link Hashers} class for additional functions.
034 * <br>
035 * The inner classes provide alternate, faster hashing algorithms. Lightning, Wisp, and Falcon
036 * have no theoretical basis or grounds in any reason other than empirical testing for why they
037 * do what they do, and this seems to be in line with many widely-used hashes (see: The Art of
038 * Hashing, http://eternallyconfuzzled.com/tuts/algorithms/jsw_tut_hashing.aspx ). That said, Wisp
039 * performs very well, ahead of Arrays.hashCode (10.5 ms instead of 15 ms) for over a million
040 * hashes of 16-element long arrays, not including overhead for generating them, while SipHash and
041 * FNV-1a take approximately 80 ms and 135-155 ms, respectively, for the same data). Lightning and
042 * Falcon perform less-well, with Lightning taking 17 ms instead of 15 ms for Arrays.hashCode, and
043 * Falcon taking about 12.3 ms but slowing down somewhat if a 32-bit hash is requested from long
044 * data. All of these have good, low, collision rates on Strings and long arrays.
045 * <br>
046 * Storm is a variant on Lightning with 64 bits for a salt-like modifier as a member variable,
047 * which can make 2 to the 64 individual hashing functions from one set of code. Mist is similar,
048 * but has 128 bits for some hashes (including any calls to hash64()) and 64 bits for some other
049 * hashes (only calls to hash() with data that doesn't involve long or double arrays). Storm and Mist
050 * have some properties of a cryptographic hash, but is not recommended it for that usage. It is,
051 * however ideal for situations that show up often in game development where end users may be able
052 * to see and possibly alter some information that you don't want changed (i.e. save data stored on
053 * a device or in the browser's LocalStorage). If you want a way to verify the data is what you
054 * think it is, you can store a hash, using one of the many-possible hash functions this can
055 * produce, somewhere else and verify that the saved data has the hash it did last time; if the
056 * exact hashing function isn't known (or exact functions aren't known) by a tampering user,
057 * then it is unlikely they can make the hash match even if they can edit it. Storm is slightly
058 * slower than Lightning just like Mist is slightly slower than Wisp, at about 20 ms for a
059 * million hashes with Storm instead of Lightning's 17 ms, and at about 18 ms for Mist for the same
060 * data instead of Wisp's 10.5, but should never be worse than twice as slow as Arrays.hashCode, and
061 * is still about three times faster than the similar SipHash that SquidLib previously had here.
062 * <br>
063 * All of the hashes used here have about the same rate of collisions on identical data
064 * (testing used Arrays.hashCode, all the hashes in here now, and the now-removed SipHash), with
065 * any fluctuation within some small margin of error. Lightning has been changed frequently but
066 * is considered stable now, but it isn't being considered to replace the FNV-1a algorithm in
067 * CrossHash for compatibility reasons. Both Lightning and Wisp seem to meet all the criteria for
068 * a good hash function, though, including doing well with a visual test that shows issues in
069 * FNV-1a and especially Arrays.hashCode. Wisp is considered stable now, and is not considered yet
070 * for replacing FNV-1a for the same reason. Chariot is not recommended. Falcon is only recommended
071 * if you need an alternate algorithm for some reason and need only 64-bit hashes; it has issues when
072 * generating 32-bit hashes of long arrays or double arrays. Mist is not considered stable yet, but
073 * probably will be soon; it offers an improvement on speed and salting over Storm, and its collision
074 * rates were very very low in testing.
075 * <br>
076 * To help find patterns in hash output in a visual way, you can hash an x,y point, take the bottom 24 bits,
077 * and use that as an RGB color for the pixel at that x,y point. On a 512x512 grid of points, the patterns
078 * in Arrays.hashCode and the default CrossHash algorithm (FNV-1a) are evident, and Sip (implementing SipHash)
079 * does approximately as well as Lightning, with no clear patterns visible (Sip has been removed from SquidLib
080 * because it needs a lot of code and is slower than Storm and especially Lightning). The idea is from a
081 * technical report on visual uses for hashing, http://www.clockandflame.com/media/Goulburn06.pdf .
082 * <ul>
083 * <li>{@link java.util.Arrays#hashCode(int[])}: http://i.imgur.com/S4Gh1sX.png</li>
084 * <li>{@link CrossHash#hash(int[])}: http://i.imgur.com/x8SDqvL.png</li>
085 * <li>(Former) CrossHash.Sip.hash(int[]): http://i.imgur.com/keSpIwm.png</li>
086 * <li>{@link CrossHash.Lightning#hash(int[])}: http://i.imgur.com/afGJ9cA.png</li>
087 * </ul>
088 * <br>
089 * Note: This class was formerly called StableHash, but since that refers to a specific
090 * category of hashing algorithm that this is not, and since the goal is to be cross-
091 * platform, the name was changed to CrossHash.
092 * Created by Tommy Ettinger on 1/16/2016.

100    public static int hash(final boolean[] data) {
101        if (data == null)
102            return 0;
103        int h = -2128831035, len = data.length - 1, o = 0;
104        for (int i = 0; i <= len; i++) {
105            o |= (data[i]) ? (1 << (i & 7)) : 0;
106            if ((i & 7) == 7 || i == len) {
107                h ^= o;
108                h *= 16777619;
109                o = 0;
110            }
111        }
112        return h;
113    }

315        if (data == null)
316            return 0;
317        int h = -2128831035, len = data.length;
318        start %= len;
319        end %= len;
320        if (end <= start || start < 0 || end <= 0)
321            return 0;
322        for (int i = start; i < end; i++) {
323            h ^= data[i] & 0xff;
324            h *= 16777619;
325            h ^= data[i] >>> 8;
326            h *= 16777619;
327        }
328        return h;

601            return 0;
602        for (int i = start; i < end; i++) {
603            h ^= data[i] & 0xff;
604            h *= 1099511628211L;
605            h ^= data[i] >>> 8;
606            h *= 1099511628211L;
607        }
608        return h;
609    }

656     * methods when the type being compared is a multi-dimensional array (which normally requires the heavyweight method
657     * {@link Arrays#deepEquals(Object[], Object[])} or doing more work yourself; this reduces the work needed to
658     * implement fixed-depth equality). As mentioned in the docs for {@link IHasher#areEqual(Object, Object)}, example
659     * code that hashes 2D char arrays can be done using an IHasher for 1D char arrays called charHasher:

661     * || ((left instanceof char[][] && right instanceof char[][])
662     * ? equalityHelper((char[][]) left, (char[][]) right, charHasher)

664     *
665     * @param left
666     * @param right
667     * @param inner
668     * @return
669     */

671        if (left == right)
672            return true;
673        if ((left == null) ^ (right == null))
674            return false;
675        for (int i = 0; i < left.length && i < right.length; i++) {
676            if (!inner.areEqual(left[i], right[i]))
677                return false;
678        }
679        return true;
680    }

1043    /**
1044     * A quick, simple hashing function that seems to have good results. Like LightRNG, it stores a state that
1045     * it updates independently of the output, and this starts at a large prime. At each step, it takes the
1046     * current item in the array being hashed, adds a large non-prime used in LightRNG's generation function
1047     * (it's 2 to the 64, times the golden ratio phi, and truncated to a signed long), multiplies by a prime
1048     * called the "state multiplier", adds the result to the state and stores it, multiplies the value of the
1049     * state by another prime called the "output multiplier", then XORs the current result with that value
1050     * before moving onto the next item in the array. A finalization step XORs the result with a complex value
1051     * made by adding the state (left over from the previous step) to what was the output multiplier, adding
1052     * the last known value for result to the phi-related constant from LightRNG, multiplying that pair, adding
1053     * the initial state (which turns out to be unusually good for this, despite no particularly special numeric
1054     * qualities other than being a probable prime) and then bitwise-rotating it left by a seemingly-random
1055     * number drawn from the highest 6 bits of the state.
1056     * <br>
1057     * This all can be done very quickly; a million hashes of a million different 16-element long arrays can be
1058     * computed in under 18-20 ms (in the benchmark, some amount of that is overhead from generating a new
1059     * array with LongPeriodRNG, since the benchmark uses that RNG's state for data, and the default
1060     * Arrays.hashCode implementation is only somewhat faster at under 16 ms). After several tries and tweaks
1061     * to the constants this uses, it also gets remarkably few hash collisions. On the same 0x100000, or
1062     * 1048576, RNG states for data, Lightning gets 110 collisions, the JDK Arrays.hashCode method gets 129
1063     * collisions, Sip (implementing SipHash) gets 145 collisions, and CrossHash (using the FNV-1a algorithm)
1064     * gets 135 collisions. Storm depends on the salt chosen, but with one initialized with the phi-based
1065     * constant that shows up in LightRNG and here, Storm gets 116 collisions. Dispersion is not perfect, but
1066     * at many bit sizes Lightning continues to have less collisions (it disperses better than the other hashes
1067     * with several quantities of bits, at least on this test data). Lightning also does relatively well, though
1068     * it isn't clearly ahead of the rest all the time, when hashing Strings, especially ones that use a larger
1069     * set of letters, it seems (FakeLanguageGen was used to make test data, and languages that used more
1070     * characters in their alphabet seemed to hash better with this than competing hashes for some reason).
1071     * <br>
1072     * There is certainly room for improvement with the specific numbers chosen; earlier versions used the
1073     * state multiplier "Neely's number", which is a probable prime made by taking the radix-29 number
1074     * "HARGNALLINSCLOPIOPEEPIO" (a reference to a very unusual TV show), truncating to 64 bits, and rotating
1075     * right by 42 bits. This version uses "Neely's number" for an initial state and during finalization, and
1076     * uses a different probable prime as the state multiplier, made with a similar process; it starts with the
1077     * radix-36 number "EDSGERWDIJKSTRA", then does the same process but rotates right by 37 bits to obtain a
1078     * different prime. This tweak seems to help with hash collisions. Extensive trial and error was used to
1079     * find the current output multiplier, which has no real relationship to anything else but has exactly 32 of
1080     * 64 bits set to 1, has 1 in the least and most significant bit indices (meaning it is negative and odd),
1081     * and other than that seems to have better results on most inputs for mystifying reasons. Earlier versions
1082     * applied a Gray code step to alter the output instead of a multiplier that heavily overflows to obfuscate
1083     * state, but that had a predictable pattern for most of the inputs tried, which seemed less-than-ideal for
1084     * a hash. Vitally, Lightning avoids predictable collisions that Arrays.hashCode has, like
1085     * {@code Arrays.hashCode(new long[]{0})==Arrays.hashCode(new long[]{-1})}.
1086     * <br>
1087     * The output multiplier is 0xC6BC279692B5CC83L, the state multiplier is 0xD0E89D2D311E289FL, the number
1088     * added to the state (from LightRNG and code derived from FastUtil, but obtained from the golden ratio
1089     * phi) is 0x9E3779B97F4A7C15L, and the starting state ("Neely's Number") is 0x632BE59BD9B4E019L.
1090     * <br>
1091     * To help find patterns in hash output in a visual way, you can hash an x,y point, take the bottom 24 bits,
1092     * and use that as an RGB color for the pixel at that x,y point. On a 512x512 grid of points, the patterns
1093     * in Arrays.hashCode and the default CrossHash algorithm (FNV-1a) are evident, and Sip (implementing
1094     * SipHash) does approximately as well as Lightning, with no clear patterns visible (Sip has been removed
1095     * from SquidLib because it needs a lot of code and is slower than Storm and especially Lightning). The
1096     * idea is from a technical report on visual uses for hashing,
1097     * http://www.clockandflame.com/media/Goulburn06.pdf .
1098     * <ul>

2791    /**
2792     * Strongly universal hashing based loosely on Daniel Lemire's (Apache-licensed) code and paper, available at
2793     * https://github.com/lemire/StronglyUniversalStringHashing . This should have among the best statistical qualities
2794     * of any of these hashes while being fairly fast. It is like {@link Storm} in that you need to instantiate a
2795     * Chariot object to use its hashing functions, but unlike Storm's single long it uses to salt the hash, these
2796     * objects have a somewhat-large, expansible cache of random numbers they use to modify every result differently
2797     * based on its position in the input array. The cache's size is related to the largest array, String, or similar
2798     * sequence that this has been required to hash, and the cache's size should be about 4 bytes per byte that needs to
