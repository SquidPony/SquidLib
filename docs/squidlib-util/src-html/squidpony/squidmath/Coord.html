007/**
008 * A 2D coordinate with (constant) x and y fields. Coord objects are immutable; a single pool of Coord values, with
009 * x and y each ranging from -3 to 255, is shared by all users of Coord. This pool helps reduce pressure on the
010 * garbage collector when many Coord values would have been created for some purpose and quickly discarded; instead
011 * of creating a new Coord with a constructor, you use the static method {@link #get(int, int)}, which retrieves an
012 * already-existing Coord from the pool if possible, and always returns a usable Coord.
013 * <br>
014 * The Coord class is a fundamental part of SquidLib; any class that uses positions on a grid makes use of it here.
015 * It finds usage naturally in classes throughout {@link squidpony.squidgrid}, with {@link squidpony.squidgrid.zone}
016 * providing an abstraction around groups of Coord and {@link squidpony.squidgrid.iterator} providing various ways to
017 * iterate through the Coords that make up a larger shape. In this package, {@link squidpony.squidmath}, a few classes
018 * should be pointed out. {@link CoordPacker} is a class with all static methods that provides various ways to compress
019 * the memory usage of regions made of many Coord values (and can be constructed in other ways but still provide Coords
020 * later), but since Coords don't use much memory anyway, the real use of the class is for manipulating the shapes and
021 * sizes of the regions those Coords are part of. {@link GreasedRegion} has similar functionality to CoordPacker, but
022 * where CoordPacker is purely static, taking and returning regions as encoded, usually-low-memory-cost arrays of
023 * {@code short} that it considers immutable, a GreasedRegion is a mutable object that allows the same region-altering
024 * techniques to be applied in-place in a way that is relatively (very) low-time-cost. If deciding between the two,
025 * GreasedRegion should usually be preferred, and CoordPacker cannot actually be used when storing regions in larger
026 * than a 256x256 space (usually when the Coord pool has been expanded; see below); GreasedRegion can store potentially
027 * large positions.
028 * <br>
029 * More on the Coord pool used by this class:  Coords can't always be retrieved from the pool; Coord.get constructs a
030 * new Coord if one of x or y is unusually large (greater than 255) or too negative (below -3). The upper limit of 255
031 * is not a hard rule; you can increase the limit on the pool by calling {@link #expandPoolTo(int, int)} or
032 * {@link #expandPool(int, int)}, which cause more memory to be spent initially on storing Coords but can save memory
033 * or ease GC pressure over the long term by preventing duplicate Coords from being created many times. The pool can
034 * never shrink because allowing that would cause completely unpredictable results if existing Coords were in use, or
035 * could easily cause crashes on Android after resuming an application that had previously shrunken the pool due to
036 * platform quirks. Long story short, you should only expand the pool size when your game needs a larger set of 2D
037 * points it will commonly use, and in most cases you shouldn't need to change it at all.
038 * 
039 * Created by Tommy Ettinger on 8/12/2015.
040 */

059    public static Coord get(final int x, final int y)
060    {
061        if(x >= -3 && y >= -3 && x < POOL.length - 3 && y < POOL[x + 3].length - 3)
062            return POOL[x + 3][y + 3];
063        else return new Coord(x, y);
064    }

066        /**
067     * Gets the angle in degrees to go between two Coords; 0 is up.
068         * @param from the starting Coord to measure from
069         * @param to the ending Coord to measure to
070         * @return The degree from {@code from} to {@code to}; 0 is up
071         */
072        public static double degrees(final Coord from, final Coord to) {
073                final int x = to.x - from.x;
074                final int y = to.y - from.y;
075                double angle = Math.atan2(y, x);
076                double degree = Math.toDegrees(angle);
077                degree += 450;// rotate to all positive and 0 is up
078                degree %= 360;// normalize
079                return degree;
080        }

333         * @return Whether {@code this} is adjacent to {@code c}. Not that a cell is
334         *         not adjacent to itself with this method.

336        public boolean isAdjacent(final Coord c) {
337                switch (Math.abs(x - c.x)) {
338                case 0:
339                        return Math.abs(y - c.y) == 1;
340                case 1:
341                        return y == c.y || Math.abs(y - c.y) == 1;
342                default:
343                        return false;
344                }
345        }

347        /**
348         * Precondition: {@code this} is {@link #isAdjacent(Coord) adjacent} to
349         * {@code adjacent}.
350     *
351         * @param adjacent
352         *            A {@link Coord} that is {@link #isAdjacent(Coord) adjacent} to
353         *            {@code this}.
354         * @return The direction to go from {@code this} to {@code adjacent} i.e.
355         *         the direction {@code d} such that {@code translate(this, d)}
356         *         yields {@code adjacent}.
357         * @throws IllegalStateException
358         *             If {@code this} isn't adjacent to {@code adjacent}.
359         */
360        /* different implementation than before, closer to isAdjacent() */
361        public Direction toGoTo(final Coord adjacent) {
362        switch (adjacent.x - x) {
363            case 0:
364                switch (adjacent.y - y)
365                {
366                    case -1: return Direction.UP;
367                    case 1: return Direction.DOWN;
368                    default:
369                        throw new IllegalStateException(this + " is not adjacent to " + adjacent);
370                }
371            case 1:
372                switch (adjacent.y - y)
373                {
374                    case -1: return Direction.UP_RIGHT;
375                    case 0: return Direction.RIGHT;
376                    case 1: return Direction.DOWN_RIGHT;
377                    default:
378                        throw new IllegalStateException(this + " is not adjacent to " + adjacent);
379                }
380            case -1:
381                switch (adjacent.y - y)
382                {
383                    case -1: return Direction.UP_LEFT;
384                    case 0: return Direction.LEFT;
385                    case 1: return Direction.DOWN_LEFT;
386                    default:
387                        throw new IllegalStateException(this + " is not adjacent to " + adjacent);
388                }
389            default:
390                throw new IllegalStateException(this + " is not adjacent to " + adjacent);
391        }
392        }

440    /**
441     * Gets the hash code for this Coord; does not use the standard "auto-complete" style of hash that most IDEs will
442     * generate, but instead uses bit mixing (differently for x and y, with each multiplied by a different large int), a
443     * XOR of the mixed x and y, a seemingly-random right shift, and an overflowing multiplication by a large prime.
444     * <br>
445     * This changed at least twice in SquidLib's history. In general, you shouldn't rely on hashCodes to stay the same
446     * across platforms and versions, whether for the JDK or this library. SquidLib (tries to) never depend on the
447     * unpredictable ordering of some hash-based collections like HashSet and HashMap, instead using its own
448     * {@link OrderedSet} and {@link OrderedMap}; if you use the ordered kinds, then the only things that matter about
449     * this hash code are that it is fast (it's fast enough) and that it doesn't collide often (which is now much more
450     * accurate than in the last version of this method).
451     * @return an int that should, for most different Coord values, be significantly different from the other hash codes
452     */
453    @Override
454    public int hashCode() {
455        int x2 = 0x9E3779B9 * x, y2 = 0x632BE5AB * y;
456        return ((x2 ^ y2) >>> ((x2 & 7) + (y2 & 7))) * 0x85157AF5;
457    }

459    /**
460     * Something like hashCode(), but reversible with {@code Coord.decode()}. Works for Coords between roughly -256 and
461     * 32000 in each of x and y, but will probably only decode to pooled Coords if x and y are both between -3 and 255
462     * (inclusive for both).
463     * @return an int as a unique code for this Coord
464     */
465    public int encode()
466    {
467        return ((x + 256) << 16) ^ (y + 256);
468    }

470    /**
471     * An alternative to getting a Coord with Coord.get() only to encode() it as the next step. This doesn't create a
472     * Coord in the middle step. Can be decoded with Coord.decode() to get the (x,y) Coord.
473     * @param x the x position to encode
474     * @param y the y position to encode
475     * @return the coded int that a Coord at (x,y) would produce with encode()
476     */
477    public static int pureEncode(final int x, final int y)
478    {
479        return ((x + 256) << 16) ^ (y + 256);
480    }
481    /**
482     * This can take an int produced by {@code someCoord.encode()} and get the original Coord back out of it. It
483     * works for all pooled Coords where the pool hasn't been expanded past about 32,000 in either dimension. It even
484     * works for Coords with negative x or y as well, if they are no lower than -256 in either dimension. This will
485     * almost certainly fail (producing a gibberish Coord that probably won't be pooled) on hashes produced by any other
486     * class, including subclasses of Coord.
487     * @param code an encoded int from a Coord, but not a subclass of Coord
488     * @return the Coord that gave hash as its hashCode()
489     */
490    public static Coord decode(final int code)
491    {
492        return get((code >>> 16) - 256, (code & 0xFFFF) - 256);
493    }

504    private static Coord[][] POOL = new Coord[259][259];
505    static {
506        int width = POOL.length, height = POOL[0].length;
507        for (int i = 0; i < width; i++) {
508            for (int j = 0; j < height; j++) {
509                POOL[i][j] = new Coord(i - 3, j - 3);
510            }

542    /**
543     * Enlarges the pool of cached Coords to the given width and height, and doesn't change
544     * a dimension if it would be reduced in size.
545     * Cached Coord values will be reused by Coord.get instead of re-allocated each time.
546     * The default pool allows Coords with x and y each between -3 and 255, inclusive, to
547     * be cached, and is considered to have width and height of 256 to begin with. Giving a
548     * width greater than 256 will allow Coords with x greater than 255 to be cached;
549     * likewise for height. If width or height is smaller than the current cache width or
550     * height, that dimension will not change, but the other still may if it is valid. You
551     * cannot shrink the pool size.
552     * @param width the new width for the pool of cached Coords; will be ignored if smaller than the current width
553     * @param height the new height for the pool of cached Coords; will be ignored if smaller than the current height
554     */
555    public static void expandPoolTo(final int width, final int height)
556    {
557        expandPool(Math.max(0, width + 3 - POOL.length), Math.max(0, height + 3 - POOL[0].length));
558    }

560    /**
561     * Enlarges the pool of cached Coords by the given amount of expansion for x and y.
562     * Cached Coord values will be reused by Coord.get instead of re-allocated each time.
563     * The default pool allows Coords with x and y each between -3 and 255, inclusive, to
564     * be cached, and this can increase the size in the positive direction. If either
565     * xIncrease or yIncrease is negative, this method returns immediately and does nothing
566     * else; the same is true of both arguments are zero. You cannot shrink the pool size.
567     * @param xIncrease the amount to increase cache's width by
568     * @param yIncrease the amount to increase cache's height by
569     */
570    public static void expandPool(final int xIncrease, final int yIncrease)
571    {
572        if(xIncrease < 0 || yIncrease < 0 || (xIncrease | yIncrease) == 0 )
573            return;
574        int width = POOL.length, height = POOL[0].length;
575        Coord[][] POOL2 = new Coord[width + xIncrease][height + yIncrease];
576        for (int i = 0; i < width; i++) {
577            POOL2[i] = new Coord[height + yIncrease];
578            System.arraycopy(POOL[i], 0, POOL2[i], 0, height);
579            for (int j = 0; j < height + yIncrease; j++) {
580                if(POOL2[i][j] == null) POOL2[i][j] = new Coord(i - 3, j - 3);
581            }
582        }
583        for (int i = width; i < width + xIncrease; i++) {
584            POOL2[i] = new Coord[height + yIncrease];
585            for (int j = 0; j < height + yIncrease; j++) {
586                POOL2[i][j] = new Coord(i - 3, j - 3);
587            }
588        }
589        POOL = POOL2;
590    }
